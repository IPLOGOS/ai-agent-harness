# Harness Deployment Guide
## How an agent deploys and lives in the harness without explicit requests

---

## The mechanism in 30 seconds

```text
┌─────────────────────────────────────────────────────────────┐
│                         ONLY ONCE                           │
│                                                             │
│  Human runs the Bootstrap Prompt                            │
│       ↓                                                     │
│  Agent reads the Playbook (like a SKILL.md)                 │
│       ↓                                                     │
│  Agent GENERATES all harness files:                         │
│    AGENTS.md, docs/*, Makefile targets, scripts/* │
│       ↓                                                     │
│  git commit — the harness is in the repo                    │
└─────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│                 ALL SUBSEQUENT SESSIONS                     │
│                                                             │
│  Claude Code / Codex opens the repo                         │
│       ↓                                                     │
│  The root map is AUTO-LOADED into context:                  │
│    • Claude Code → CLAUDE.md + .claude/settings.json hooks  │
│    • Codex → AGENTS.md                                      │
│  (native behavior — no explicit request)                    │
│       ↓                                                     │
│  The agent follows the harness rules:                       │
│    - Code Navigation Rule                                   │
│    - Loop Breaker                                           │
│    - Definition of Done                                     │
│    - Golden Principles                                      │
│    - Merge Gates                                            │
│    - Tool hierarchy                                         │
│       ↓                                                     │
│  The user just says "fix the auth bug"                      │
│  → the agent operates WITHIN the harness automatically      │
└─────────────────────────────────────────────────────────────┘
```

**Why it works without an explicit request:**
Claude Code automatically loads `CLAUDE.md` at each session, then executes the `.claude/settings.json` hooks (SessionStart, PreToolUse, PostToolUse, Stop).
Codex automatically loads `AGENTS.md` for each task.
Cursor automatically loads `.cursorrules` or `AGENTS.md`.

The harness IS the agent's default context. The user doesn't have to ask for anything.
For Claude Code, the hooks add a layer of mechanical enforcement on top of the instructions.

## Step 1: Prepare the source files

(Note: this is not a formal "skill" in the Codex sense — it's a plain reference file.)

## Step 2: The Bootstrap Prompt (idempotent)

The prompt is designed to be **re-run as many times as necessary** — on a fresh repo, a repo without a harness, a partial harness, or an outdated harness. It never breaks what works.

**The 4-phase mechanism:**

```text
PHASE 0 — AUDIT     → detects current state, produces a gap report
PHASE 1 — PLAN      → classifies each element: CREATE / UPDATE / SKIP
PHASE 2 — EXECUTE   → applies only necessary changes
PHASE 3 — VERIFY    → version stamp, tests, commit
```

**Versioning:** The playbook has a version number (e.g., `3.4.0`). The bootstrap writes this version to `.harness/harness.meta.json`. On each run, it compares the installed version against the playbook version to detect necessary upgrades.

The complete prompt is in `BOOTSTRAP_PROMPT.md`. Here are the shortcuts:

### First deployment (existing repo):

```bash
# Copy the universal prompt from BOOTSTRAP_PROMPT.md and paste it into Claude Code
# The agent will: audit → plan → execute → verify → commit
```

### Update after playbook upgrade:

```bash
claude "Read .harness/PLAYBOOK_SKILL.md. The playbook has been updated. \
EXECUTION MODE: review. \
Run all phases. Upgrade the harness to match the new version. \
Preserve all repo-specific content. Only add missing [REQUIRED] elements."
```

### Audit only (verify without modifying):

```bash
claude "Read .harness/PLAYBOOK_SKILL.md. EXECUTION MODE: audit. \
Run PHASE 0 only. Show the gap report. Do not modify anything."
```

### New project:

```bash
claude "Read .harness/PLAYBOOK_SKILL.md. \
This is a NEW repository. Run all phases including the Initializer (§1.5). \
The project is: [DESCRIPTION]. Stack: [STACK]. Domains: [DOMAINS]."
```

### The 5 handled scenarios:

| Scenario | PHASE 0 detects | Result |
| :--- | :--- | :--- |
| Fresh repo | FRESH | Generates everything, tailored to the project |
| Existing repo without harness | HAS_PARTIAL_HARNESS | Creates harness files, preserves existing compliant targets |
| Harness v3.3, playbook v3.4.0 | HAS_HARNESS, upgrade needed | Adds only what is missing from v3.4.0 |
| Compliant and up to date harness | COMPLIANT | "No changes needed" — STOP |
| Partial harness (AGENTS.md yes, MERGE_GATES no) | HAS_PARTIAL_HARNESS | Fills the gaps, doesn't touch what works |

The complete prompt with the 4 phases is located in **`BOOTSTRAP_PROMPT.md`**.

## Step 3: Verify the harness is auto-active

After the bootstrap, the repo also contains `.harness/harness.meta.json`:

```json
{
  "playbook_version": "3.4.0",
  "last_audit": "2026-03-12",
  "last_audit_commit": "a1b2c3d",
  "state": "COMPLIANT",
  "items_created": ["AGENTS.md", "docs/ARCHITECTURE.md", "..."],
  "items_updated": [],
  "items_skipped": []
}
```

This file allows the next audit to detect the installed version and update only what has changed.

After the bootstrap, the repo contains:


### Verification test:

Open a NEW Claude Code session in the repo and type:

```text
fix the auth bug in src/auth/handler.ts
```

**Without mentioning the harness.** The agent should automatically:
1.  Read AGENTS.md (auto-loaded)
2.  Execute `make project-summary` to orient itself
3.  Use `make find-refs SYMBOL=handleAuth` instead of `cat **/*.ts`
4.  Respect the Loop Breaker (max 3 cycles)
5.  Create proof artifacts
6.  Verify merge gates before proposing a merge

If the agent does all this without being asked → the harness is successfully deployed.

## Step 4: Claude Code vs. Codex Architecture

### Claude Code (TARGET: claude-code) — CLAUDE.md is the primary

```text
CLAUDE.md                    ← AUTO-LOADED — this is the primary
AGENTS.md                    ← Codex mirror (generated from CLAUDE.md)
.claude/
├── settings.json            ← hooks + permissions (mechanical enforcement)
├── agents/                  ← specialized subagents (initializer, verifier, etc.)
├── skills/                  ← on-demand capabilities
└── commands/                ← slash commands (/audit-harness, /init-task)
src/<domain>/CLAUDE.md       ← per-domain overrides
```

**Why CLAUDE.md is primary:** Claude Code natively loads `CLAUDE.md`.
The hooks in `.claude/settings.json` transform text-based rules into mechanical enforcement.
The subagents in `.claude/agents/` are specialists with restricted tools and isolated context.
See `CLAUDE_CODE_BLUEPRINT.md` for the complete implementation guide.

### Codex (TARGET: codex) — AGENTS.md is the primary

```text
AGENTS.md                    ← AUTO-LOADED — this is the primary
CLAUDE.md                    ← Claude Code mirror (generated from AGENTS.md)
docs/                        ← all enforcement is in docs/ and Makefile
```

Codex has no equivalent to `.claude/settings.json` or `.claude/agents/`.
Enforcement remains in CI gates and Make targets.

### Compatibility table

| Tool | Primary | Native surfaces | Mechanical enforcement |
| :--- | :--- | :--- | :--- |
| Claude Code | `CLAUDE.md` | `.claude/settings.json`, `.claude/agents/`, `.claude/skills/` | Hooks (SessionStart, PreToolUse, PostToolUse, Stop) |
| Codex | `AGENTS.md` | None beyond the filesystem | CI + Make targets |
| Cursor | `.cursorrules` | Limited | Verify by version |
| Aider | `.aider.conventions` | Limited | Verify by version |

> **Note:** Claims about Cursor and Aider may drift between versions.
> See Appendix F of the playbook for dated details.

## Step 5: Harness evolution (without explicit prompt)

The harness maintains itself via automations defined in the playbook:

| Automation | Trigger | Effect |
| :--- | :--- | :--- |
| **docs-gardener** | Daily 09:00 | Scans docs/ for staleness, opens fix-up PRs |
| **golden-principle-sweep** | Daily 10:00 | Scans for violations of GOLDEN_PRINCIPLES.md |
| **quality-score-update** | Weekly on Monday 08:00 | Updates QUALITY_SCORE.md |
| **lessons-promotion** | Weekly on Friday 16:00 | Review of accumulated lessons → promotion to golden principles, hooks, or verifier rules |
| **harness-simplification** | After each model upgrade | §2.7: tests each component with canary tasks, removes what is no longer load-bearing |
| **dream-sweep** | Auto: 24h + 5 sessions (dual gate) | §1.8: consolidates agent memory — merges, prunes, indexes under 200 lines |
| **Harness self-fix** | On every agent failure | §2.3: "prompt the agent to build its own new guardrail" |

To activate these automations in Codex:

```bash
# Example: docs-gardener as a Codex automation
codex automate --schedule "0 9 * * *" \
  --prompt "Read AGENTS.md. Scan docs/ for stale documentation. \
            For each stale doc, open a fix-up PR. \
            Do not invent — only correct based on what the code does."
```

## Summary: the complete chain

```text
Human installs the playbook → .harness/PLAYBOOK_SKILL.md
                ↓
Human runs the Bootstrap Prompt (ONCE)
                ↓
Agent generates AGENTS.md + docs/* + scripts/* + Makefile targets
                ↓
git commit "chore: scaffold harness"
                ↓
═══════════════════════════════════════
FROM HERE ON, EVERYTHING IS AUTOMATIC:
═══════════════════════════════════════
                ↓
Session N: user says "fix bug X"
  → CLAUDE.md auto-loaded (Claude Code) / AGENTS.md auto-loaded (Codex)
  → agent follows the harness natively
                ↓
Session N+1: user says "add feature Y"
  → same AGENTS.md, same rules
  → progress.json ensures continuity
                ↓
Automations: docs-gardener, golden-sweep, quality-score, lessons-promotion
  → harness maintains itself without intervention
                ↓
When an agent fails:
  → Loop Breaker → human escalation
  → human asks the agent to fix the harness (§2.3)
  → canary tasks verify the fix doesn't regress
  → harness evolves incrementally
```

## FAQ

**Q: Does the user need to mention the harness in every session?**
No. The root map is auto-loaded. Claude Code loads `CLAUDE.md` (+ executes the `.claude/settings.json` hooks). Codex loads `AGENTS.md`. This is the native behavior of each tool.

**Q: How do I update the harness?**
Modify AGENTS.md or docs/* directly, or ask the agent: "Read .harness/PLAYBOOK_SKILL.md and update docs/MERGE_GATES.md to add Gate 9 for..."

**Q: What if I switch models (Claude → GPT → Gemini)?**
AGENTS.md is tool-agnostic. The harness works with any agent supporting the format. Verify with canary tasks after the switch.

**Q: Is the 1400-line playbook injected into every session?**
No. Only AGENTS.md (< 100 lines) is injected. The playbook stays in `.harness/` as a reference skill, read only during bootstrap or updates.

**Q: How do I adapt this for an existing repo with legacy code?**
The bootstrap prompt analyzes the existing repo and adapts. Add to the prompt: "This is an existing repo with [X] lines of code. Analyze the current structure before scaffolding. Preserve existing conventions where they don't conflict with the harness."

**Q: How does harness observability work?**
Each hook emits structured events to `.claude/traces/run-<uuid>.jsonl` (one file per session). Events carry a correlated `run_id`, standardized failure codes (e.g., `VERIFY_HEAD_MISMATCH`, `TOOL_DENIED_MERGE`), and git state. The verifier produces a full evidence bundle. The `make metrics-*` targets compute evaluations from these traces. This is Level 1 (implementable now). Level 2 (centralized OTel spans) requires an external wrapper.

**Q: How do I validate a harness change?**
Re-run the corpus of canary tasks defined in `docs/CANARY_TASKS.md` (5 representative tasks). Compare retries, failure codes, verifier pass rate, and time. If there is a regression > 20% on 3+ tasks → revert the change.

**Q: When should I use the multi-agent architecture (planner + generator + evaluator)?**
For complex, lengthy builds (>1h, >10 features). The planner turns a short prompt into a full spec. The generator builds sprint by sprint. The evaluator actively exercises the application (Playwright MCP) and grades against negotiated criteria. For small fixes, a single agent is enough. See §2.7 of the playbook.

**Q: How do agents communicate?**
Via files in `.claude/handoffs/`: spec.md (planner → generator), sprint-N-contract.md (generator ↔ evaluator), sprint-N-feedback.md (evaluator → generator), session-N-handoff.md (context reset). Files are committed to git for auditability.

**Q: Compaction or context reset?**
It depends on the model. If the model shows "context anxiety" (terminates too early when context is large), use context resets with structured handoffs. If the model handles long contexts well, compaction is sufficient. Test both on your canary tasks. Anthropic found that Sonnet 4.5 required resets while Opus 4.5/4.6 handled compaction well.

**Q: How do I simplify the harness after a model upgrade?**
Simplification protocol (§2.7): test each component with and without via canary tasks. If removing it doesn't degrade results → remove the component. Tag what remains as MODEL-WORKAROUND with a last-tested date.

**Q: How does agent memory work?**
The agent has two memory levels: CLAUDE.md (human instructions, < 200 lines) and `.claude/memory/` (agent memory, auto-captured). During sessions, the agent writes corrections, patterns, and decisions into topic files. Between sessions, the `dream-sweep` subagent consolidates: merges duplicates, deletes obsolete facts, converts relative dates to absolute, and keeps the MEMORY.md index under 200 lines. Consolidation triggers automatically after 24h AND 5 sessions (dual gate). This design is compatible with Claude Code Auto Dream when it ships natively, and works with other tools via the harness.

**Q: What is the memory promotion chain?**
Session observation → topic file in `.claude/memory/` → consolidation via dream-sweep → after 5 occurrences of a pattern, promotion to `docs/generated/lessons.md` → after validation, promotion to golden principle, hook, or verifier rule. Every link in the chain is traceable.
